# 엔진 렌더러 제약 (실측으로 확정)

폰트를 소비하는 렌더러가 여럿이고, 각각이 폰트블록을 다르게 취급한다.
아래 규칙은 추정이 아니라 **실기 GS 덤프 해독과 크래시 재현으로 확정**한 것이다.

## 렌더러 지도

| 렌더러 | 대상 화면 | 폰트 소스 | 특성 |
|---|---|---|---|
| 셸 렌더러 | 브리핑/메뉴/시스템 | 셸·캡션 폰트 13종의 **파일 range 테이블을 읽음** | 라이브 코드 치환 시 즉시 반영 |
| 캡션 렌더러 | 미션 무전 자막/목표 배너 | 캡션 대형 6종(id120000x) | 아래 제약 전부 적용 |
| HUD/포즈 렌더러 | 미션 HUD·포즈 메뉴 | 원본 레이아웃 기준 **baked 매핑** | 글리프 재배열(rebuild) 시 깨짐 |
| 경보 렌더러 | 중앙 경보 배너 | idx4614 (id1200011) | charset이 좁음(0x5F 없음) |

## 캡션 렌더러의 폰트 테이블 취급 (전부 실측)

1. **range 테이블에 엔트리를 append하면 절대 발견되지 않는다** → 해당 코드는 fallback `？`.
   (0x20 append, 0x22 append 두 차례 실증 — GS 덤프에서 `？` 글리프 UV 확인)
2. **range 테이블을 정렬 재배열해 다시 쓰면 폰트 로드 시 게임이 죽는다** (2회 재현).
   테이블에는 시작코드 순서 외의 순서 의존성이 있다.
3. **기존 엔트리의 값(gi) 재작성과 텍스처(글리프 픽셀) 편집은 안전하다.**
   따라서 주입은 "기존 코드의 박스에 덮어그리기"로만 한다.
4. 0x20(반각 공백)은 테이블에 넣어도 렌더되지 않음(엔진 선처리 추정).

**결론: 폰트블록 구조·순서는 1바이트도 바꾸지 말 것. 텍스처만 편집한다.**

## GS 덤프 역추적 방법 (검증 도구)

PCSX2의 GS 덤프(.gs.zst)를 파싱하면 "엔진이 실제로 어떤 텍스처의 어떤 UV를 샘플링했는지"를
역산할 수 있다. 캡션 wrong-glyph/？ 원인 진단에 결정적이었다.

- 덤프 파서: `parse_gsdump_gif.py` (v8 state, packet/GIF 태그 순회)
- 캡션 draw는 `prim=6(sprite), FST=0` — 텍스처 좌표가 **UV 레지스터가 아니라 ST(float)**.
  packed ST의 레이아웃은 S@0, T@4, **Q@8** (RGBAQ 패킷에는 Q가 없음에 주의).
- 텍셀 = (S/Q × 2^TW, T/Q × 2^TH). 이 값을 파일 메트릭 박스와 대조하면
  글자당 "엔진이 고른 글리프"가 나온다 — 파일 테이블과의 불일치 즉시 검출.
- 업로드 payload와 ISO 엔트리의 바이트 비교로 "어느 엔트리가 로드됐는가"도 확정 가능.

## 텍스트 인코딩 제약

- 문장부호를 2바이트 코드로 인코딩하면 텍스트가 커져 5만 문자열 컨테이너가
  오버플로 폴백된다(실측). **반각 부호는 1바이트 유지**가 필수.
- 공백도 동일 — 전각 공백(2B) 전면 사용 시 라디오 섹션 402곳 오버플로,
  1바이트 가짜 공백 사용 시 4곳으로 감소.
